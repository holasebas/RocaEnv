CEM (Componente Elemento Modificador)
Componente
En el sigueinte codigo HTML se define un componente conformado por elementos al que se nombro "lego". El nombrado CSS para los componentes dentro del layout debe ser en minusculas, sin ñ´s, carcteres especiales, espacios, guiones y guines bajos. el uso de CamelCase es opcional (miComponente)
                
<div class="lego">
    <div class="lego_head">
    <div class="lego_body">
    <div class="lego_legs">
</div>
Elemento:
A su vez el componente "lego" cuenta con elementos que lo conforman "head", "body" y "legs". Los Elementos cuentan con las mismas reglas de nombrado que los componentes. Sin embargo los elementos cuentan con un carateristica diferente la cual es; le prescede el nombre del componente al que pertenece en este caso es "lego" y sus elemetos seran separado exclusivamente por un guion bajo (_) lego_head 
                
<div class="lego">
    <div class="lego_head">
</div>
                
               
                
<div class="lego">
    <div class="lego_body">
</div>
                
               
                
<div class="lego">
    <div class="lego_legs">
</div>
Modificador:
Un componente tanto un elemento pueden contar con modificadores enlcual como su nombre lo indica modifica cierta propiedad, estos modificadores cuentan con las mismas caracteristicas de nombrado que un componente o elemento con su diferencia de que le prescede el nombre del componente o elemento al cual esta modificado seguido de doble guio (--) logo_head--fireman 
                
<div class="lego">
    <div class="lego-head lego_head--pirat">
</div>

                
               
                
<div class="lego">
    <div class="lego-head lego_head--goblin">
    <div class="lego-body lego-body--red">
    <div class="lego-legs">
</div>
Como se menciono anteriormente, los Componentes tambien pueden contar con modificadores; lego--woman 
                
<div class="lego lego--woman">
    <div class="lego-head lego_head--goblin">
    <div class="lego-body lego-body--red">
    <div class="lego-legs">
</div>
Utilerias
Componentes y Elementos tambien pueden estar conformados por utilirias, con las mismas carateristicas de nombrado; (minusculas, sin acentos sin ñ's o cualquier otro caracter especial) estas utilerias seran nombradas bajo el mismo esquema de Bootstrap (mx-auto, p-1, text-center etc.) name-name. Ej. bg-tomato el cual cambia el color del background de un componente o elemento,

Nota: esta propiedad que defininos como "bg-tomato" es una utileria no un modificador y hay que saber identificar una de otra y cuando deberia definir una utileria y cuando un modificador. Dendera de las carateristica de nuestro componente. 
                
bg-tomato{
    background-color:tomato;
}
<div class="lego lego--woman bg-tomato">
    <div class="lego-head lego_head--goblin>
    <div class="lego-body lego-body--red">
    <div class="lego-legs">
</div>
RESUMEN
componente: todo aquel nombrado que no contenga ningun tipo de guion o espaciado es un componente.
${componente}_elemento: todo aquel nombrado que separado por un guion (_) es un elemeto
${elemento o componente} --elemento: todo aquel nombrado separado por doble guion(--) es un modificador de determinado componente o elemento
nombre-nombre: todo aquel nombrado separado por un guion simple(-) es una utileria
LAYOUT VS CEM
El Maquetado(layout) y nombrado de clases (CEM) son independientes y nada tienen que ver uno de otro, es decir; el maquetado por lo general es construido con base utilerias, mayormente todas provendran de Bootsrap mientras que el nombrado de componentes es con base a CEM + utilirias que pueden ser propias o provenientes de bootstrap

UI/COMPONENT GLOSARY
Dentro de cada proyecto es estrictamente necesario, tener un UI glosary, que en pocas palabras es lo equivalente a un APIdoc en el Front-End. UI glosary es una vista o ruta privada uiglosary_view que contiene el listadado de todo componente o elemento con cada uno de sus respectivos modificadores hayan sido definidos el proyecto. 
            
    function UIGlosaryView(){
        return(
            <div class="cta"></div>
            <div class="cta cta--primary"></div>
            <div class="cta cta--big"></div>
            <div class="cta cta--big cta--primary"></div>
            <CardComponent />
            <OtroReactComponent />
        )
    }
La principal utilidad del UI glosary es poder tener un panorama general de la aplicacion en cuanto a Componentes y CSS, es decir; si tengo un requerimiento el cual podria ser:

"El boton de 'comprar ahora' debe estar brincando."

Dado el requerimiento anterior, partiriamos desde el UIGlosary y ver si existe ya un modificador que cumpla con el requerimiento, o crearlo en caso de no haberlo, esto para seguir con el patron CEM y evitar malas practicas o redundancia en nuestra hoja de estilos, cta--bounce el cual modificara el comportamiento de nuestro componente cta.
RESPONSIVE BREAKPOINTS
Los siguientes breakpoints son estrictamente requeridos y en ese mismo orden, junto con algun otro que sea necesario definir en el orden jerarquico correspondiente Ej. si se require definir un brekpoint de 340px quedaria colocado debajo del de 576px. Estos breakpoints llevan una relacion 1:1 con las utilirias de layout de Bootstrap lo cual nos permite llevar una sincronia entre estilos y layout siguiendo el esquema de cascada de css; es decir las propiedades definidas en cada brekpoint no ocurriran hasta que el breakpoint hayan sido alcanzado. 
            
@media (max-width: 1200px) { ...dispositvos extra largos }

@media (max-width: 992px) { ...dispositvos largos }

@media (max-width: 768px) { ...dispositvos medianos }

@media (max-width: 576px) { ...dispositvos pequeños }
